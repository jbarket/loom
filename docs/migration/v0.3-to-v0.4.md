# Migration guide: loom v0.3.x → v0.4

**Who this is for:** Anyone running loom v0.3.1 (the Qdrant-backed
release) who wants to move to v0.4 (sqlite-vec + fastembed).

If you're setting up loom for the first time, ignore this document.
Go to the [README](../../README.md) and run `npx github:jbarket/loom install`.

---

## What changed

v0.4 replaces the entire external-services stack with a single file:

| Component | v0.3.x | v0.4 |
|-----------|--------|------|
| **Memory storage** | Qdrant collection (running daemon) | `memories.db` (SQLite, in context dir) |
| **Embeddings** | Ollama or OpenAI (configurable) | fastembed BGE-small-en-v1.5 (local, ~33 MB ONNX, CPU-only) |
| **Env selectors** | `LOOM_MEMORY_BACKEND`, `LOOM_EMBEDDING_PROVIDER` | Gone — one stack, no selector |
| **External services required** | Qdrant + Ollama (or OpenAI API key) | None |

Your memories, refs, UUIDs, TTLs, and metadata all carry over exactly.

> **Real-world data point:** The production migration (414 Qdrant
> points → `memories.db`) completed in a few minutes, produced a
> ~2.5 MB file, and lost zero records. See
> [`docs/rescue-notes-2026-04-19.md`](../rescue-notes-2026-04-19.md)
> for the full log.

## License change: MIT → AGPL-3.0-or-later

v0.3.x was released under MIT. Starting with v0.4.0-alpha.1, loom is
licensed under **AGPL-3.0-or-later**.

What this means in practice:

- **Bundling loom into a larger product** — allowed. The AGPL applies
  to loom itself, not your product's code.
- **Forking and going proprietary** — not allowed. Forks must remain
  open-source under the AGPL.
- **Running a modified loom and letting others interact with it over
  a network** — requires you to offer the corresponding source to
  those users.
- **Personal use, self-hosted, airgapped** — no copyleft obligations
  triggered by running it privately.

If you forked v0.3.x under MIT and want to stay on that license,
your fork is fine as-is. The AGPL applies to v0.4+ releases.

## Before you start

You need:

- Qdrant still running and reachable (you need it running to export)
- Node.js ≥ 20
- The loom repo cloned locally (or use `npx github:jbarket/loom` as
  shown below)
- Your agent's context directory path (`~/.config/loom/<agent>` by
  default)

The migration script reads your Qdrant collection, re-embeds every
memory using fastembed, and writes a fresh `memories.db`. It **will
not overwrite** an existing `memories.db` — it aborts with a warning
instead of silently clobbering data.

## Step 1: Dry run

Run the script with `--dry-run` to see what it would do without
writing anything:

```bash
npx tsx scripts/migrate-qdrant-to-sqlite.ts --dry-run
```

If your Qdrant is on a non-default URL or uses a non-default
collection name:

```bash
LOOM_QDRANT_URL=http://my-qdrant:6333 \
LOOM_QDRANT_COLLECTION=my_collection \
npx tsx scripts/migrate-qdrant-to-sqlite.ts --dry-run
```

Expected output:

```
[migrate] Qdrant: http://localhost:6333/collections/loom_memories
[migrate] SQLite: /home/you/.config/loom/art/memories.db
[migrate] DRY RUN — no writes
[migrate] Fetched 414 points from Qdrant
[migrate] Initializing fastembed (first-run downloads ~33MB)...
[migrate] Embedder ready (384-dim)
[migrate] embedded 414/414
[migrate] Written: 414 | Skipped (missing fields): 0
[migrate] Final memories table count: 414
[migrate] Final vec_memories count: 414
[migrate] DRY RUN complete. No file written.
```

If `Skipped` is non-zero, those records are missing `ref`, `title`,
or `category` fields — they won't survive the migration. Note the
count; these are the records you'll want to check in Qdrant before
proceeding.

---

## Step 2: Run the migration

When the dry run looks right, run it for real:

```bash
npx tsx scripts/migrate-qdrant-to-sqlite.ts
```

Or with custom paths:

```bash
LOOM_QDRANT_URL=http://my-qdrant:6333 \
LOOM_QDRANT_COLLECTION=my_collection \
LOOM_SQLITE_DB_PATH=~/.config/loom/art/memories.db \
npx tsx scripts/migrate-qdrant-to-sqlite.ts
```

The first run downloads the fastembed ONNX model (~33 MB) to
`~/.cache/loom/fastembed/`. Subsequent runs are instant.

The script:

1. Scrolls all points from Qdrant (paginated, 100 at a time)
2. Batch-embeds the content using fastembed (32 records per batch)
3. Inserts into SQLite inside a transaction per batch
4. Verifies the final counts and exits

---

## Step 3: Verify round-trip

### Count check

Compare the counts from the migration output against Qdrant:

```bash
# Check what Qdrant says
curl -s http://localhost:6333/collections/loom_memories | jq '.result.points_count'

# Check what SQLite says
sqlite3 ~/.config/loom/art/memories.db "SELECT COUNT(*) FROM memories;"
```

They should match (minus any records that were skipped for missing
fields).

### Spot-check recall

Start loom against the new DB and do a test recall:

```bash
LOOM_CONTEXT_DIR=~/.config/loom/art \
npx github:jbarket/loom recall "something you know should be in memory"
```

Pick 2–3 queries covering different categories (`user`, `project`,
`self`, etc.) and confirm you're getting plausible results.

### Structural check

```bash
sqlite3 ~/.config/loom/art/memories.db <<'SQL'
SELECT category, COUNT(*) as n FROM memories GROUP BY category ORDER BY n DESC;
SQL
```

This gives you a category breakdown to sanity-check against what
you'd expect.

## Step 4: Update your v0.4 config

Remove these env vars from your config — they're no longer used:

```
# DELETE THESE — v0.4 ignores them
LOOM_QDRANT_URL
LOOM_QDRANT_COLLECTION
LOOM_OLLAMA_BASE_URL
LOOM_EMBEDDING_PROVIDER
LOOM_MEMORY_BACKEND
```

The only env vars that matter in v0.4:

| Variable | Default | Notes |
|----------|---------|-------|
| `LOOM_CONTEXT_DIR` | `~/.config/loom/default` | Path to your agent's context dir |
| `LOOM_SQLITE_DB_PATH` | `<context>/memories.db` | Override DB path (optional) |
| `LOOM_FASTEMBED_MODEL` | `fast-bge-small-en-v1.5` | Change only if you need different dims |
| `LOOM_FASTEMBED_CACHE_DIR` | `~/.cache/loom/fastembed/` | ONNX model cache |
| `LOOM_CLIENT` | *(unset)* | Harness hint: `claude-code`, `gemini-cli`, etc. |
| `LOOM_MODEL` | *(unset)* | Model hint: `claude-opus`, `gemma4`, etc. |

See [`.env.example`](../../.env.example) for a copy-pasteable reference.

## Step 5: New v0.4 directories (optional but recommended)

v0.4 introduces three new subdirectories in the context dir:

```
$LOOM_CONTEXT_DIR/
├── harnesses/    # per-harness manifests (tool prefixes, delegation primitives)
├── models/       # per-model manifests
└── procedures/   # procedural-identity docs (6 seed templates ship with loom)
```

These are optional — loom works without them. But if you want the
full v0.4 setup:

```bash
# Adopt the six procedural-identity seed templates
npx github:jbarket/loom procedures adopt --all \
  --context-dir ~/.config/loom/art

# Scaffold a harness manifest for your AI harness
npx github:jbarket/loom harness init claude-code \
  --context-dir ~/.config/loom/art
```

Or run `npx github:jbarket/loom install` and let the setup skill
handle all of this interactively.

## What to do with your old Qdrant collection

The migration script doesn't touch Qdrant — your collection stays as
it was. Once you've verified the SQLite migration looks right, you
have three options:

1. **Leave it** — harmless, just taking up space. Safe choice if
   you're not sure yet.
2. **Delete the collection** — `DELETE http://localhost:6333/collections/loom_memories`.
   Do this only after a few days of using the new DB and confirming
   recall quality.
3. **Stop the Qdrant daemon entirely** — if loom was its only
   consumer, you no longer need it running.

There is no automated rollback — if you delete the collection and
find a problem, you'd need to rerun the migration from a backup.
Don't skip Step 3.

## Troubleshooting

**"SQLite DB already exists at ... Aborting"**

The target `memories.db` already exists. Either:
- You've already migrated — check if `memories.db` looks populated
  with `sqlite3 memories.db "SELECT COUNT(*) FROM memories;"`
- You have an existing (empty) db from an aborted run — delete it
  and rerun

**"Qdrant scroll failed (404)"**

The collection name doesn't match. Check with:
```bash
curl http://localhost:6333/collections
```
Then set `LOOM_QDRANT_COLLECTION` to the right name.

**"Qdrant scroll failed (connection refused)"**

Qdrant isn't running, or is on a different port. Set `LOOM_QDRANT_URL`.

**fastembed download stalls or fails**

The first run downloads ~33 MB from Hugging Face. On a slow or
firewalled connection, set `LOOM_FASTEMBED_CACHE_DIR` to a directory
on a volume with write access, or pre-download the model manually.

**Skipped count is unexpectedly high**

Records missing `ref`, `title`, or `category` are skipped. These
were likely corrupted or written by an old version of loom. Check:
```bash
curl -s -X POST http://localhost:6333/collections/loom_memories/points/scroll \
  -H 'Content-Type: application/json' \
  -d '{"with_payload":true,"limit":100}' | jq '.result.points[] | select(.payload.ref == null)'
```
