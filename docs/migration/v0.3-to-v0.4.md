# Migration: v0.3.x → v0.4

This guide is for users who were running loom v0.3.1 (Qdrant + Ollama era) and
want to move to v0.4 (sqlite-vec + fastembed, zero external services).

## What changed

| | v0.3.1 | v0.4 |
|---|---|---|
| **Memory store** | Qdrant (Docker daemon) | `memories.db` (single SQLite file) |
| **Embeddings** | Ollama (local daemon) or OpenAI API | fastembed ONNX (in-process, ~33MB download on first run) |
| **npm package** | `loom` (unpublished) | `loomai` |
| **External services** | Qdrant at `localhost:6333`, Ollama at `localhost:11434` | None |
| **License** | MIT | AGPL-3.0-or-later |

## Before you start

Back up your Qdrant collection. The migration script is read-only by default with
`--dry-run`, but there is no undo once you run it for real.

```bash
# Verify Qdrant is running and accessible
curl -s http://localhost:6333/collections/loom_memories | jq .

# Check how many points you have
curl -s http://localhost:6333/collections/loom_memories | jq '.result.points_count'
```

## Step 1: Install v0.4

```bash
# From npm (alpha.7 was the first published release):
npm install -g loomai

# Or from source:
git clone https://github.com/jbarket/loom.git
cd loom && npm ci && npm run build
```

## Step 2: Dry-run the migration

The migration script (`scripts/migrate-qdrant-to-sqlite.ts`) reads every point from
your Qdrant collection, re-embeds the content using fastembed, and inserts into a
new `memories.db`. Run `--dry-run` first to verify the connection and count:

```bash
# From an npm install:
npx tsx node_modules/loomai/scripts/migrate-qdrant-to-sqlite.ts --dry-run

# From a source clone:
npx tsx scripts/migrate-qdrant-to-sqlite.ts --dry-run
```

Default env:

| Variable | Default |
|---|---|
| `LOOM_QDRANT_URL` | `http://localhost:6333` |
| `LOOM_QDRANT_COLLECTION` | `loom_memories` |
| `LOOM_SQLITE_DB_PATH` | `~/.config/loom/art/memories.db` |

Override any of these if your setup differs:

```bash
LOOM_QDRANT_URL=http://myhost:6333 \
LOOM_QDRANT_COLLECTION=my_collection \
LOOM_SQLITE_DB_PATH=~/.config/loom/myagent/memories.db \
npx tsx scripts/migrate-qdrant-to-sqlite.ts --dry-run
```

The dry-run output shows point count, embedder dimensions, and what would be written.
No file is created.

## Step 3: Run the migration

```bash
# WARNING: aborts if memories.db already exists — move or delete it first
npx tsx scripts/migrate-qdrant-to-sqlite.ts
```

The script:
1. Reads all points from Qdrant (paginated, handles large collections)
2. Downloads the fastembed BGE-small model (~33MB, once, cached at
   `~/.cache/loom/fastembed/`)
3. Embeds content in batches of 32
4. Inserts into `memories.db` with WAL mode
5. Prints a final count — verify it matches your Qdrant point count

A skipped count > 0 means some Qdrant points were missing `ref`, `title`, or
`category` fields. Those are gaps in the original data, not migration errors.

## Step 4: Verify round-trip

```bash
# Count check
loom memory list --json | jq 'length'

# Spot-check recall
loom recall "something you know is in your memories"

# Doctor check
loom doctor --json | jq '.existingAgents[].hasMemoriesDb'
```

## Step 5: Stop Qdrant and Ollama (optional)

v0.4 does not connect to either service. You can stop them, pause their Docker
containers, or leave them running — loom will not touch them.

```bash
# Docker example
docker stop qdrant
docker stop ollama
```

If you want to keep the Qdrant data as a backup, leave the container stopped but
don't delete its volume yet. Once you're satisfied with the migration, you can
remove it.

## Step 6: Update your MCP config

The server key and tool prefix are unchanged (`loom`, `mcp__loom__*`), but the
startup command changes if you were running loom via a source path:

```json
{
  "loom": {
    "command": "npx",
    "args": ["loomai"],
    "env": {
      "LOOM_CONTEXT_DIR": "/path/to/your/context"
    }
  }
}
```

Run `loom inject --all` to let loom rewrite the managed sections in your harness
dotfiles to the current format:

```bash
loom inject --all --context-dir ~/.config/loom/art
```

## License change

v0.3.1 was released under **MIT**. v0.4.0-alpha.1 and all subsequent releases are
**AGPL-3.0-or-later**.

What this means:
- **Using loom for your own agents** — unchanged. Personal use, internal company
  tooling, and running loom as a local MCP server are all fine.
- **Forking and modifying loom** — if you modify loom and expose it over a network
  (e.g., as a hosted service or API), you must offer the source of your modifications
  to users of that service.
- **Bundling loom into a larger product** — allowed. Closing the loom-derived code
  and reselling it as a service is not.
- **v0.3.1 remains MIT** — if you need MIT for a specific reason, pin to v0.3.1 and
  don't migrate.

The AGPL was chosen because loom's core value is the agent's persistent identity —
infrastructure that should stay open.

## Troubleshooting

**"SQLite DB already exists"** — move or delete the existing `memories.db` first.
The script will not overwrite it. Back it up if it has data you care about.

**fastembed download fails** — the first run downloads ~33MB of ONNX model weights
to `~/.cache/loom/fastembed/`. Ensure you have internet access. Set
`LOOM_FASTEMBED_CACHE_DIR` to a writable path if the default location is read-only.

**Qdrant connection refused** — make sure Qdrant is running and the `LOOM_QDRANT_URL`
matches. The migration needs read access to the collection; it never writes to Qdrant.

**Skipped > 0 but count looks right** — points missing `ref`, `title`, or `category`
were in an inconsistent state in Qdrant. These are data gaps, not migration failures.
Check the original Qdrant point payloads if you need to recover them.
